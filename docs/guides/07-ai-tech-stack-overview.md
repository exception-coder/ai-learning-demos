## 二、技术栈详解

### 模型层 — 大模型基础

| 技术 | 是什么 | 做什么用 | 使用环节 |
|------|--------|---------|---------|
| **大模型原理** | GPT、DeepSeek 等大语言模型的底层原理 | 理解 Transformer、注意力机制、Token 等概念 | 最底层基础，所有 AI 应用的根基 |
| **提示词工程** | 设计有效的 Prompt 让大模型按预期输出 | 控制模型行为、提高输出质量、减少幻觉 | 贯穿所有环节，每次调用大模型都在用 |
| **模型微调** | 在预训练模型基础上用自有数据继续训练 | 让通用模型适配特定领域（如医疗、法律） | 当通用模型在你的业务场景表现不够好时 |

### 框架层 — 开发工具链

| 技术 | 是什么 | 做什么用 | 使用环节 |
|------|--------|---------|---------|
| **Spring AI** | Spring 生态的 AI 开发框架（Java） | 统一对接各种大模型，提供 ChatClient、Tool、RAG 等抽象 | Java 后端开发 AI 应用的首选框架 |
| **LangChain4j** | Java 版 LangChain，AI 应用开发框架 | 提供链式调用、Agent、RAG 等能力，与 Spring AI 定位类似 | 另一个 Java AI 框架选项，社区生态不同 |
| **Java** | 主力开发语言 | 企业级 AI 应用开发 | 后端服务、MCP Server、Agent 逻辑 |
| **Python** | AI 领域最主流的语言 | 模型训练、微调、快速原型、数据处理 | 模型微调、数据预处理、部分 Agent 框架 |

**Spring AI vs LangChain4j 怎么选？**

```
Spring AI      → Spring 生态用户首选，与 Spring Boot 无缝集成
LangChain4j    → 功能更丰富，社区更活跃，不依赖 Spring
两者可以共存   → 在同一个项目中混合使用
```

### 能力层 — 让 AI 能"做事"

| 技术 | 是什么 | 做什么用 | 使用环节 |
|------|--------|---------|---------|
| **FunctionCall** | 让大模型调用外部函数的机制 | 模型识别意图 → 生成函数调用参数 → 你的代码执行 → 结果返回模型 | AI 需要查数据库、调 API、执行计算时 |
| **MCP** | Model Context Protocol，标准化 AI 连接外部系统的协议 | 让不同 AI 应用复用同一套工具/数据/提示词 | 需要标准化对接多个外部系统时 |
| **Skill** | 独立的能力单元 | 封装一个具体能力（查天气、算数学、搜索） | Agent 内部的能力模块 |
| **RAG** | Retrieval Augmented Generation，检索增强生成 | 先从知识库检索相关文档，再让模型基于文档回答 | 企业知识问答、客服、文档搜索 |
| **多模态 RAG** | 支持图片、音频、视频等多种格式的 RAG | 不仅检索文字，还能检索图片表格等 | 知识库包含图片、PDF、表格时 |
| **上下文工程** | 系统化管理送给模型的上下文信息 | 决定给模型看什么、不看什么、怎么组织 | 提高模型回答质量，控制 Token 成本 |

**FunctionCall vs MCP vs Skill 的关系：**

```
FunctionCall → 底层机制：模型如何告诉你"我要调用某个函数"
MCP          → 连接协议：标准化暴露这些函数给 AI 应用
Skill        → 组织方式：把相关函数打包成一个能力单元

一个完整的调用链：
Agent → 选择 Skill → Skill 内部通过 FunctionCall 调用 Tool
                      ↑
                  MCP 协议暴露 Tool 清单
```

### 编排层 — 让 AI 能"思考"

| 技术 | 是什么 | 做什么用 | 使用环节 |
|------|--------|---------|---------|
| **Agent** | 能自主规划和执行任务的 AI 程序 | 理解意图 → 规划步骤 → 调用工具 → 返回结果 | 需要 AI 自主完成多步骤任务时 |
| **ReactAgent** | React 模式的 Agent（Reasoning + Acting） | 思考→行动→观察→再思考，循环直到完成 | 最常用的 Agent 模式，适合大部分场景 |
| **ReflectionAgent** | 带自我反思能力的 Agent | 执行后自我评估，发现问题则修正重来 | 需要高质量输出时（如代码生成、文案撰写） |
| **PlanExecuteAgent** | 先规划再执行的 Agent | 先生成完整计划，再逐步执行 | 复杂任务（如 DeepResearch、多步分析） |
| **Human in the Loop** | 人机协作模式 | 关键节点暂停，等人类确认后继续 | 高风险操作（如退款、删除、发布） |
| **Workflow** | 工作流引擎，预定义执行流程 | 用可视化方式编排 AI 调用顺序 | 固定流程的自动化（如审批、数据处理） |
| **n8n** | 开源的工作流自动化平台 | 可视化拖拽编排 AI + API + 数据库等节点 | 快速搭建 AI 自动化流程，不需要写太多代码 |
| **AgentScope** | 阿里开源的多 Agent 框架（Python） | 多个 Agent 协作完成复杂任务 | 需要多个 AI 角色协作的场景 |
| **长期记忆** | 让 Agent 记住历史交互信息 | 跨对话保持上下文，记住用户偏好 | 需要持续服务同一用户的场景 |

**四种 Agent 模式对比：**

```
ReactAgent         → 边想边做，灵活但可能走弯路
ReflectionAgent    → 做完回头看，质量高但更慢
PlanExecuteAgent   → 先想好再做，适合复杂任务
Human in the Loop  → 关键节点问人，安全但打断流程

实际项目中常常混合使用：
PlanExecuteAgent（先规划）
  → ReactAgent（逐步执行每个子任务）
    → Human in the Loop（关键步骤人工确认）
      → ReflectionAgent（最终结果自查）
```

### 应用层 — 实际产品

| 项目 | 用到哪些技术 | 核心挑战 |
|------|-------------|---------|
| **企业内部智能客服** | RAG + 多模态RAG + RAG测评 | 检索准确率、回答质量评估 |
| **企业级端到端通用Agent** | Agent + DeepResearch + Workflow | 多步骤规划、工具编排、结果生成（PPT/网页） |
| **旅行规划助手** | Agent + FunctionCall + 长期记忆 | 多轮对话、个性化推荐、外部 API 调用 |
| **仿PDD自动退款** | Agent + FunctionCall + Human in the Loop | 自动填表、风控审核、异常处理 |
| **舆情分析助手** | n8n + RAG + Agent | 数据采集、情感分析、自动化报告 |
| **mini claude code** | Agent + FunctionCall + ReflectionAgent | 代码理解、代码生成、自我检查 |

## 三、学习路径建议

```
第一阶段：地基（你已经在这里了）
  大模型原理 → 提示词工程 → Spring AI → FunctionCall
                                         ↓
第二阶段：能力
  MCP → Skill → RAG → 上下文工程
                        ↓
第三阶段：编排
  ReactAgent → ReflectionAgent → PlanExecuteAgent
  → Human in the Loop → Workflow/n8n → 长期记忆
                                         ↓
第四阶段：实战项目
  智能客服（RAG 为主）→ 通用 Agent → 复杂业务场景
                                         ↓
第五阶段：进阶（可选）
  模型微调 → AgentScope（多 Agent）→ 自定义框架
```
